# ADR-0001: Информационная Архитектура Документации

Статус: `Accepted`

## Контекст

Документация APIDev исторически росла как набор частично пересекающихся архитектурных заметок, process-guides и продуктовых описаний. Это привело к дублированию, языковым расхождениям, конфликтам по каноническим терминам и отсутствию явной карты источников истины.

## Решение

Принята жесткая иерархия документации:

- каждая тема имеет один canonical source;
- документы разделяются по статусам `Authoritative`, `Reference`, `Guide`, `Draft / Vision`, `Historical`;
- human-facing документы в `docs/` и `CONTRIBUTING.md` ведутся на русском языке;
- каноническая команда генерации в документации — `apidev gen`;
- старая документация либо мигрируется в новую структуру, либо удаляется после переноса уникального содержания.

Принята следующая верхнеуровневая структура:

- `docs/product/`
- `docs/architecture/`
- `docs/process/`
- `docs/reference/`
- `docs/decisions/`

Канонические entrypoints:

- `docs/README.md`
- `docs/architecture/README.md`
- `CONTRIBUTING.md`

## Последствия

Положительные:

- снижается риск противоречивых норм;
- участникам проще находить нужный документ;
- архитектурные, process и reference материалы становятся явно разведены;
- появляется стабильная база для дальнейших ADR и repository-wide policies.

Негативные и компромиссы:

- первый большой рефакторинг документации создает крупный diff;
- требуется дисциплина по поддержанию source-of-truth map;
- часть исторических документов придется удалить или архивировать.

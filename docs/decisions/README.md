# Архитектурные Решения APIDev

Статус: `Authoritative`

Этот каталог хранит ADR для долгоживущих архитектурных и process-решений, которые должны быть воспроизводимы и ссылочно устойчивы.

## Когда создавать ADR

Создавайте ADR, если решение:

- меняет архитектурные границы или layering;
- вводит новое repository-wide правило;
- меняет информационную архитектуру документации;
- фиксирует долгоживущую policy по generated/manual boundary, CLI contract или process;
- supersede-ит существующее решение.

## Формат

- Имя файла: `ADR-XXXX-<slug>.md`
- Нумерация: четырехзначная, монотонно возрастающая
- Один ADR = одно решение

## Рекомендуемая структура ADR

- Заголовок
- Статус
- Контекст
- Решение
- Последствия
- Связанные документы

## Superseded policy

Если решение заменяет старое:

- новый ADR должен явно ссылаться на старый;
- старый ADR остается в истории и не удаляется;
- в старом ADR добавляется отметка, что он superseded.

# Глоссарий APIDev

Статус: `Authoritative`

Этот документ закрепляет каноничные термины проекта. Один термин должен использоваться в одном значении; альтернативные формулировки допустимы только если они не создают неоднозначности.

## Термины

### `authoritative document`

Нормативный документ, который является единственным источником истины по теме. В остальных документах допускается только summary и ссылка на него.

### `reference document`

Справочный документ, который раскрывает тему через примеры, детали или таблицы, но не вводит новые нормативные правила.

### `contract`

Файл API-описания, который задает декларативное описание endpoint-а: method, path, auth, request/query shape, response и errors.

### `operation_id`

Стабильный идентификатор операции, вычисляемый из относительного пути контрактного файла и уникальный в рамках проекта.

### `diff`

Предпросмотр planned changes между желаемым generated output и текущим состоянием файлов без записи на диск.

### `drift`

Расхождение между generated output, который должен быть получен из текущих контрактов и шаблонов, и уже сохраненными generated-файлами.

### `check mode`

Режим проверки без записи файлов. Для APIDev это в первую очередь `apidev gen --check`.

### `generated`

Код или артефакты, которыми владеет генератор и которые допускается безопасно пересоздавать.

### `manual`

Код или артефакты, которыми владеет разработчик и которые не должны перезаписываться генератором.

### `bridge layer`

Ручной слой интеграции между generated transport-кодом и ручной бизнес-логикой: registry, auth wiring, error mapping и другие glue concerns.

### `tool runtime`

Runtime самого APIDev как CLI-инструмента: загрузка контрактов, diff, validate, render, write/check.

### `target app runtime`

Runtime целевого приложения, использующего generated output APIDev вместе с manual-логикой.

### `architecture rule`

Нормативное ограничение на зависимости, слои, side effects, границы записи, языковую политику или другие архитектурные инварианты.

## Правила использования терминов

- В нормативных документах использовать именно `apidev gen` как каноническое имя команды генерации.
- Если документ говорит о будущем состоянии, использовать `target state`, а не смешивать его с `current state`.
- Если термин уже описан здесь, не вводить рядом новый конкурирующий термин без явного объяснения.

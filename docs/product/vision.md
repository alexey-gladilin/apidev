# Продуктовое Видение APIDev

Статус: `Draft / Vision`

Этот документ описывает продуктовую рамку APIDev, его целевое назначение и границы ответственности. Он не описывает обязательное текущее поведение инструмента и не подменяет архитектурные правила или roadmap.

## Контекст

APIDev развивается как инструмент contract-driven разработки API в семействе dev-tools. Его задача — уменьшить объем повторяющейся transport-работы вокруг API, сохранив бизнес-логику под ручным контролем команды.

## Проблема

В типичных API-проектах повторяется один и тот же набор технических слоев:

- сигнатуры роутов и HTTP-метаданные;
- request/response-схемы;
- стандартные ошибки и OpenAPI metadata;
- шаблонный wiring зависимостей;
- базовые contract-тесты.

Это повышает стоимость старта новых endpoint-ов и создает риск рассинхронизации между контрактом, transport-слоем и тестами.

## Гипотеза ценности

Если APIDev будет генерировать стабильные контрактные и transport-артефакты, а изменчивое поведение оставит ручному коду, команда получит:

- более быструю поставку endpoint-ов;
- единообразные API-контракты и OpenAPI;
- безопасную перегенерацию;
- более предсказуемый CI-процесс;
- упрощенный онбординг в проект.

## Граница ответственности

### Что APIDev генерирует

- transport-скелет на основе контрактов;
- operation map и route wrappers;
- request/response/error schema templates;
- OpenAPI-метаданные;
- шаблоны для generated-only артефактов;
- контрактно-ориентированные тестовые заготовки в следующих фазах развития продукта.

### Что APIDev не генерирует

- бизнес-правила и use-case логику;
- repository internals и SQL-логику;
- политику авторизации уровня домена;
- произвольные side-effectful integrations;
- доменно-специфичную интерпретацию ошибок.

Принцип: APIDev должен генерировать стабильные интерфейсы и повторяющиеся слои, но не изменчивое поведение.

## Текущее состояние

Сейчас APIDev уже умеет:

- инициализировать структуру проекта;
- загружать YAML-контракты;
- проверять базовые правила валидации, включая уникальность `operation_id`;
- строить diff без записи файлов;
- генерировать deterministic skeleton для generated-зоны;
- проверять drift через `apidev gen --check`.

## Целевое состояние

Целевое развитие продукта:

- более строгая семантическая валидация контрактов;
- генерация полноценного transport-слоя;
- более формальная compatibility policy;
- расширяемость шаблонов без fork основного генератора;
- опциональная интеграция с `dbspec` как read-only источником schema hints.

## Модель runtime

Для APIDev важны две разные runtime-зоны:

- `tool runtime` — собственно CLI-инструмент, который читает контракты, строит diff, рендерит шаблоны и пишет generated-артефакты;
- `target app runtime` — код в целевом приложении, который использует generated transport и ручную бизнес-логику.

## Интеграция с DBSpec

Интеграция с `dbspec` рассматривается как опциональная и read-only. APIDev не должен становиться владельцем DB schema artifacts и не должен требовать обязательную связку с `dbspec` для базового workflow.

## Что не является целью

- превращение APIDev в full-stack application generator;
- генерация бизнес-логики;
- замена архитектурной дисциплины проекта самим генератором;
- смешение generated и manual кода в одной зоне владения.

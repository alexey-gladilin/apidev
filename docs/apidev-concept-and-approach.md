# apidev: Концепт и подход для дальнейшего анализа

## 1. Контекст и цель

`apidev` планируется как часть семейства **dev-tools** и должен следовать общим конвенциям (независимый пакет, независимый CLI, скрытая рабочая директория, опциональные интеграции).

Цели высокого уровня:

- сделать разработку Web API более декларативной и contract-driven;
- сократить повторяющийся backend-скелет;
- сохранить бизнес-логику явной и поддерживаемой;
- интегрироваться с `dbspec` для ускорения поставки сервисов end-to-end.

Ключевая идея:

- аналитик/архитектор описывает API-контракты в декларативной спецификации;
- `apidev` генерирует стабильный каркас кода по шаблонам;
- разработчики реализуют бизнес-поведение в негенерируемых точках расширения;
- перегенерация безопасна и идемпотентна при изменении контрактов.

## 2. Формулировка проблемы

В типичных FastAPI-сервисах много повторяющихся слоев:

- сигнатуры роутов и HTTP-метаданные;
- декларации request/response-схем;
- стандартные ошибки и OpenAPI-документация;
- шаблонный dependency injection;
- контрактные endpoint-тесты.

Эта повторяемость замедляет разработку и повышает риск несогласованности между доменами.

Одновременно полная генерация бизнес-логики нежелательна, потому что доменное поведение часто меняется и требует ручного владения.

## 3. Продуктовая гипотеза

Если `apidev` будет генерировать контрактно-ориентированные и повторяющиеся слои, оставляя бизнес-логику в ручном владении, команды получат:

- более быструю поставку endpoint-ов;
- единообразные контракты и документацию;
- более безопасную эволюцию за счет детерминированной перегенерации;
- снижение стоимости онбординга и поддержки.

В связке с `dbspec` ценность растет, так как схема БД и API-контракты становятся связанными артефактами, а не изолированными документами.

## 4. Граница ответственности: что генерировать, а что оставлять ручным

### 4.1 Генерировать (Да)

`apidev` должен генерировать:

- декларации endpoint-роутов (method/path/params/status/response model);
- классы request/response/error-схем;
- стандартные OpenAPI-метаданные ответов;
- регистрацию роутеров и модульный каркас;
- шаблоны контрактных API-тестов (smoke + ключевые error path);
- опционально типизированные клиентские стабы (future phase).

### 4.2 Оставлять ручным (Нет)

`apidev` не должен генерировать:

- бизнес-правила/use-case-логику;
- SQL/оптимизацию запросов и внутренности repository-слоя;
- реализацию политик авторизации (RBAC/ABAC детали);
- внешние интеграции с нетривиальными side effect;
- доменно-специфичную интерпретацию ошибок.

Принцип:

- **генерировать стабильные интерфейсы, а не изменчивое поведение**.

## 5. Разделение на runtime `apidev` и runtime целевого приложения

Чтобы избежать двусмысленности, фиксируем две разные runtime-зоны:

- **runtime инструмента `apidev` (внутри бинарника/CLI)**:
  - загрузка и валидация контрактов;
  - планирование генерации;
  - рендер шаблонов;
  - diff/check-режимы.
- **runtime целевого приложения (часть сгенерированного проекта)**:
  - тонкий слой стыковки generated-кода и ручной бизнес-логики;
  - реестр `operation_id -> handler`;
  - базовый маппинг ошибок и подключение auth dependency.

Критично: это **не сервер внутри `apidev`**, а код, который живет в приложении разработчика после генерации.

### 5.1 Минимальный паттерн стыковки для MVP

Рекомендуемый паттерн подключения бизнес-логики:

- сгенерированные роуты вызывают стабильный handler-интерфейс по `operation_id`;
- ручные реализации регистрируются в реестре;
- generated-код не импортирует напрямую изменчивую доменную реализацию.

Преимущества:

- безопасность перегенерации;
- явные ошибки при отсутствии handler-а;
- простая подмена/mocking в тестах;
- нет ручного патчинга generated route-файлов.

### 5.2 Предлагаемое разделение слоев в целевом приложении

- Generated слой (`generated/`): контракты, route-wrapper-ы, transport-модели.
- Manual слой (`manual/handlers`, `manual/services`, `manual/repositories`): бизнес-логика и доступ к данным.
- Bridge слой (`manual/runtime`): registry, mapping ошибок, auth dependencies.

## 6. Стратегия обработки ошибок

Обработка ошибок должна быть гибридной:

- **ручная централизованная политика** в `manual/runtime` (единый источник истины);
- **декларативное описание контрактных ошибок** в спецификации endpoint-а.

Предлагаемый механизм:

- бизнес-код выбрасывает типизированные доменные исключения;
- централизованный handler маппит их в HTTP error envelope;
- генератор добавляет error-метаданные и ссылки на схемы в OpenAPI.

Обоснование:

- консистентное внешнее поведение API;
- гибкость для внутренних изменений домена;
- меньше дублирования в endpoint-коде.

## 7. Стратегия аутентификации и авторизации

Auth должен декларативно описываться в контрактах, но реализовываться вручную в bridge-слое целевого приложения (`manual/runtime`).

Примеры деклараций на уровне спецификации:

- `auth: public`
- `auth: bearer`
- `auth: bearer+scopes([...])`

Поведение генератора:

- подключение нужной dependency и OpenAPI security-описания.

Ручное поведение:

- валидация токена, извлечение principal, проверки прав.

Так generated-код остается универсальным и не хардкодит environment-specific auth-логику.

## 8. Целевой workflow использования инструмента

### 8.1 Базовый сценарий

1. Разработчик инициализирует рабочую директорию: `apidev init`.
2. Аналитик/архитектор формирует API-контракты в `.apidev/contracts/*.yaml`.
3. Контракты коммитятся в Git как проектные артефакты.
4. Разработчик запускает:
  1. `apidev validate`
  2. `apidev diff`
  3. `apidev generate`
5. Разработчик реализует бизнес-логику в `manual`-слое приложения.
6. При изменении контракта выполняется безопасная перегенерация только `generated`-файлов.

### 8.2 Встраивание в существующий и новый проекты

#### Существующий проект

Использовать поэтапное внедрение:

- начать с одного доменного модуля;
- генерировать только изолированное `generated`-поддерево;
- подключить generated-router в существующее приложение;
- не трогать текущие ручные модули;
- сравнить поведение и масштабировать постепенно.

#### Новый проект

`apidev init` создает базовую структуру:

- рабочая директория `.apidev`;
- конфигурация проекта;
- стартовый контракт;
- целевая структура generated/manual кода;
- стартовые тесты.

### 8.3 Структура рабочей директории `.apidev`

```text
.apidev/
├── config.toml
├── contracts/
│   ├── billing/
│   │   ├── create_invoice.yaml
│   │   └── get_invoice.yaml
│   └── auth/
│       └── login.yaml
├── templates/
└── .gitignore
```

Назначение директорий и файлов:

- `config.toml` — основной конфигурационный файл `apidev` для проекта (пути к контрактам и целевым директориям генерации, режимы валидации/совместимости).
- `contracts/` — API-контракты (YAML/JSON), которые являются входом генератора; это ключевой проектный артефакт, который должен версионироваться в Git.
- `templates/` — Jinja-шаблоны генерации целевых файлов. Разработчик может адаптировать шаблоны под свой проект, изменения также версионируются в Git.
- `.gitignore` — локальные исключения для временных файлов/артефактов, если они появятся в процессе работы.

Рекомендации по Git:

- Коммитить: `contracts/`, `templates/`, `config.toml` (если команда использует общий проектный конфиг).
- Не коммитить: локальные временные файлы и служебные артефакты, если они появятся.

### 8.4 Предлагаемая структура `config.toml` (MVP)

Для MVP фиксируем минимальный конфиг. По мере развития инструмента допускается расширение структуры без изменения базового сценария.

```toml
version = "1"

[contracts]
dir = ".apidev/contracts"

[generator]
generated_dir = "src/api/generated"

[templates]
dir = ".apidev/templates"
```

## 9. Модель перегенерации (критично)

Перегенерация должна быть детерминированной и безопасной:

- одинаковая входная спецификация => одинаковые выходные файлы;
- перезаписываются только generated-файлы;
- ручные файлы всегда сохраняются;
- стабильный `operation_id` вычисляется из относительного пути endpoint-файла и обязан быть уникален по проекту;
- добавление/изменение/удаление операций видно в diff до применения.

Рекомендуемый CLI-процесс:

1. `apidev validate`
2. `apidev diff`
3. `apidev generate`

Опциональные safety-режимы:

- `--check` для CI-проверки дрейфа генерации;
- `--fail-on-breaking` для совместимости контрактов.

## 10. Формат источника контракта

Рекомендация для MVP:

- основной формат: YAML;
- источник контрактов: все `*.yaml` в `.apidev/contracts/**` (без единого `api.yaml`);
- одна папка = домен, один файл = один endpoint-контракт;
- `operation_id` явно не хранится в файле, а вычисляется из относительного пути файла;
- обязательность поля задается на уровне самого поля через `required` (по умолчанию `true`);
- формат должен покрывать request/response, query params, описание endpoint и описание полей, а также ошибки с body.

Правило вычисления `operation_id`:

- вход: `<domain>/<file_stem>.yaml`
- результат: `<domain>_<file_stem>`
- пример: `billing/create_invoice.yaml` -> `billing_create_invoice`

Минимальный формат endpoint-файла (MVP):

```yaml
method: POST
path: /v1/invoices
auth: bearer # public | bearer

summary: Создать счет
description: Создает новый invoice для клиента.

query:
  - name: dry_run
    type: boolean
    required: false
    description: Только валидация без фактического создания.

request:
  body:
    type: object
    description: Данные для создания счета.
    properties:
      customer_id:
        type: string
        required: true
        description: Идентификатор клиента.
      amount:
        type: number
        required: true
        description: Сумма счета.
      currency:
        type: string
        required: true
        description: Код валюты (ISO 4217).

response:
  status: 201
  body:
    type: object
    description: Результат успешного создания.
    properties:
      invoice_id:
        type: string
        required: true
        description: Идентификатор созданного счета.
      status:
        type: string
        required: true
        description: Текущий статус счета.

errors:
  - code: VALIDATION_ERROR
    http_status: 400
    body:
      type: object
      properties:
        error_code:
          type: string
          required: true
          description: Машинный код ошибки.
        message:
          type: string
          required: true
          description: Текст ошибки для клиента.
        details:
          type: object
          required: false
          description: Дополнительные детали ошибки.

  - code: UNAUTHORIZED
    http_status: 401
    body:
      type: object
      properties:
        error_code:
          type: string
          required: true
        message:
          type: string
          required: true
```

Правило для object-схем в MVP:

- `required` у свойства опционален и трактуется как `true`, если не указан явно;
- `required: false` указывается только для реально опциональных полей.

Минимальные обязательные поля для MVP:

- `method`
- `path`
- `auth`
- `summary`
- `description`
- `response.status`
- `response.body`
- `errors[]` (каждая ошибка: `code`, `http_status`, `body`)

## 11. Интеграция с dbspec

Принцип интеграции:

- `dbspec` остается SSOT для дизайна БД;
- `apidev` остается SSOT для API-контрактов;
- интеграция опциональна и read-only.

Что `apidev` может импортировать из `dbspec`:

- подсказки скалярных типов;
- nullability;
- reference/enumeration-ограничения;
- канонические имена полей;
- кандидаты для filter/sort-полей.

Ожидаемый эффект:

- меньше рассинхронов между БД и API-моделями;
- быстрее создание CRUD/read endpoint-ов;
- проще согласованная эволюция схемы и API.

## 12. Слоистая архитектура инструментов (Onion Model)

Предлагаемая ментальная модель семейства dev-tools:

- `dbspec` — базовый слой data-contract (модели БД и структурные ограничения).
- `apidev` — слой API-контрактов и transport-логики поверх `dbspec`.
- `uidev` — слой UI-интеграции поверх `apidev` (service client, query-слой, runtime-валидация).

### 12.1 Направление зависимостей

Разрешенное направление зависимостей:

- `uidev -> apidev -> dbspec`

Запрещенное направление:

- `dbspec` не должен зависеть от `apidev` или `uidev`;
- `apidev` не должен зависеть от `uidev`.

Это сохраняет четкие границы абстракций и предотвращает циклические потоки генерации.

### 12.2 Владение контрактами по слоям

- `dbspec` владеет контрактами схемы БД.
- `apidev` владеет HTTP/API-контрактами.
- `uidev` владеет контрактами клиентского потребления API.

Каждый слой может read-only потреблять артефакты нижнего слоя и трансформировать их в собственные контракты.

### 12.3 Практические последствия

- изменения нижних слоев предсказуемо распространяются вверх;
- перегенерация возможна по слоям с понятной зоной влияния;
- команды могут внедрять инструменты инкрементально без full-stack lock-in;
- каждый инструмент остается автономным, но поддерживает единую pipeline-модель.

## 13. Ключевые риски и меры снижения

### 13.1 Риски

- template lock-in и сложность изменения generator-output;
- сложные merge/conflict-сценарии при перегенерации;
- чрезмерная генерация, приводящая к плохо читаемому коду;
- рассинхрон между декларированным контрактом и runtime-поведением.

### 13.2 Меры снижения

- строгая изоляция generated/manual слоев;
- стабильная идентичность операций через `operation_id`;
- детерминированные шаблоны и форматирование;
- генерация контрактных тестов по умолчанию;
- compatibility-check в CI;
- явное versioning схемы спецификации.

## 14. Предложение по MVP

MVP должен быть намеренно узким.

Скоуп MVP:

- инициализация проекта (`apidev init`) с созданием `.apidev` и базового `config.toml`;
- валидация контракта (`apidev validate`);
- генерация кода (`apidev generate`);
- предпросмотр изменений (`apidev diff`);
- генерация routes + schemas + базовых OpenAPI-ответов + handler-интерфейсов;
- минимальный bridge-слой приложения: registry + error mapping + auth dependency hooks;
- ручной workflow регистрации handler-ов;
- опциональный `dbspec`-импорт для базового mapping типов.

Вне MVP:

- полноценный policy-engine для authz;
- продвинутые async/workflow-скелеты для очередей задач;
- multi-language generators;
- полное reverse-engineering legacy-кода.

## 15. Критерии успешности для дальнейшей оценки

Технические сигналы:

- существенно сокращено время на endpoint scaffolding;
- перегенерация без повреждения manual-файлов;
- меньше дефектов из-за несоответствия API-контрактов;
- стабильные CI-проверки дрейфа спецификации/генерации.

Продуктовые сигналы:

- разработчики доверяют generated-структуре;
- аналитики могут предсказуемо предлагать изменения контрактов;
- ускоряется онбординг.

## 16. Открытые вопросы для следующей итерации

- точная форма DSL контракта для фильтров, пагинации и сортировки;
- политика совместимости для breaking-изменений контракта;
- стратегия жизненного цикла депрекации и вывода операций;
- глубина генерации тестов (только smoke или более широкая матрица);
- насколько строгой должна быть связка dbspec-to-apidev в MVP.

## 17. Рекомендуемые следующие шаги

1. Определить схему контракта `apidev` v1 (YAML) и правила валидации.
2. Утвердить layout `.apidev`, generated/manual директорий и политику перезаписи.
3. Реализовать маленький вертикальный срез:
  - один домен,
  - две операции,
  - generated route+schema,
  - ручная реализация handler-а.
4. Добавить `apidev diff` и CI-режим проверки.
5. Протестировать интеграцию с одной `dbspec`-сущностью и замерить снижение трудозатрат.

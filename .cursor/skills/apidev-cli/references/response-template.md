# Шаблон ответа агента для apidev-команд

Использовать компактную структуру отчета после запуска `@apidev` или авто-подобранных команд.

## Формат

```markdown
CLI version:
1. `apidev <detected-version>` (detected via `apidev --version`)

Выполнено:
1. `apidev validate --json`
2. `apidev diff --json`

Результат:
1. [Короткий статус по каждой команде]
2. [Найденные проблемы/изменения]

Требует подтверждения:
1. `apidev gen` (запись изменений в файлы)
2. `apidev init --force` (перезапись init-managed файлов)

Следующий шаг:
1. [Самый безопасный и конкретный]
```

## Правила подачи результата

- Не выводить сырой лог целиком, если нет отдельного запроса.
- Всегда указывать обнаруженную версию CLI (`apidev --version`) в начале отчета.
- Ошибки показывать с конкретной причиной и командой, на которой они возникли.
- Если ошибка указывает на отсутствие команды `apidev`, предлагать/использовать fallback `uv run apidev ...`.
- Для успешных проверок сообщать, что именно проверено (`validate`, `diff`, `gen --check`).
- Для изменений в файловой системе явно помечать, что это write-операция (`gen`, `init --force`).
- Для `init` дополнительно фиксировать выбранный профиль (`--runtime`, `--integration-mode`, `--integration-dir`), если он задан.

## Мини-контракт JSON-анализа

Если команда запускалась с `--json`, в отчете всегда отражать:

1. `summary`:
   - `status` (`ok`/`failed`)
   - `errors` (число)
   - `warnings` (число)
2. `diagnostics`:
   - список ключевых диагностик (код, severity, краткая причина)
3. `compatibility` (для `diff`/`gen`, если присутствует):
   - policy (`warn`/`strict`)
   - counts по категориям

## Политика отчета об ошибках

- При `exit code != 0` писать, что пайплайн остановлен.
- После ошибки `validate`/`diff` не предлагать `gen` как немедленный следующий шаг.
- После ошибки `gen`/`init` предлагать верификацию состояния через `validate` и `diff`.

# Решения и компромиссы

## DEC-01: Добавить root `request` как first-class часть контракта
- Решение: `request` становится нормативным root-блоком рядом с `response/errors`.
- Почему: request-часть должна быть описана и валидируема в той же модели истины.
- Компромисс: контракты с неформальным/свободным request-описанием потребуют адаптации.

## DEC-02: Сохранить strict-schema подход и запрет unknown fields
- Решение: неизвестные поля в `request` и подблоках запрещены.
- Почему: это поддерживает предсказуемость и детерминизм генерации.
- Компромисс: меньше гибкости для ad-hoc расширений без изменения спецификации.

## DEC-03: Явная проверка согласованности `request.path` с route template
- Решение: `request.path` и `{param}` в `path` обязаны совпадать по набору.
- Почему: устраняет двусмысленность между runtime path и контрактной схемой.
- Компромисс: даже «почти корректные» контракты не проходят без полного соответствия.

## DEC-04: Опциональность root-блока `request`
- Решение: root `request` опционален для операций любого HTTP-метода, если входные данные отсутствуют; при наличии `{param}` в `path` блок `request.path` обязателен.
- Почему: отсутствие входных параметров возможно не только для `GET`, поэтому обязательность `request` должна быть привязана к фактическим данным, а не к методу.
- Компромисс: нужна явная fail-fast политика для placeholder-путей без `request.path`, чтобы сохранить детерминизм без эвристик.

## DEC-05: `operation_map` как канонический слой request metadata
- Решение: path/query/body fragments публикуются в operation metadata и используются downstream.
- Почему: единый источник request-данных для transport и OpenAPI снижает расхождения.
- Компромисс: повышается связность между моделью операции и шаблонами генерации.

## DEC-06: Расширить OpenAPI projection из полной модели операции
- Решение: OpenAPI проецирует request/response/errors из единой контрактной модели.
- Почему: документация должна соответствовать фактическому transport-контракту.
- Компромисс: любые ошибки в request-модели блокируют генерацию OpenAPI (fail-fast вместо деградации).

## DEC-07: Не вводить эвристики для request-проекции
- Решение: `parameters` и `requestBody` формируются только из explicit contract metadata.
- Почему: эвристики снижают воспроизводимость и усложняют диагностику.
- Компромисс: пользователь обязан полно и явно описывать request в контракте.

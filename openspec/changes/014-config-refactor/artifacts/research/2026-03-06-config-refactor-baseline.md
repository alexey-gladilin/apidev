# Research: baseline для рефакторинга конфигурации apidev

## Scope
- Текущая модель `.apidev/config.toml`.
- Точки чтения конфигурации в runtime.
- Риски при переходе на строгий контракт без backward compatibility.

## Наблюдения
- Конфигурация уже частично секционирована (`contracts`, `generator`, `templates`, `openapi`, `evolution`).
- Поле `version` не несет полезной runtime-нагрузки и может быть источником ложной совместимости.
- Основной риск refactor-а — рассинхронизация между `init`, loader-валидацией и документацией.

## Подтвержденные факты
- Конфиг читается через TOML loader с Pydantic-валидацией.
- Release-state путь валидируется отдельно и ограничивается project-root boundary.
- Генератор и diff-контур зависят от корректного разрешения output/scaffold путей.

## Вывод
Для безопасного refactor-а нужен единый канонический контракт секций и строгая fail-fast политика на любые legacy-ключи.
